这里是linux基础，用于快速入门命令行界面，可以先移步附录查看环境配置

1. 命令行基础

手册：RTFM really helps. 使用 man 指令可以获取一个指令（或工具/函数）的手册。让我们从 man man 开始。

命令行常用命令有：ls, rm, rmdir, makedir, cd, mv, cp, cat, echo, sudo 等，功能请自行查阅手册.

1.1 流与重定向、管道

<, >, |, 2>

程序通常获取stdin（键盘到shell的输入）作为输入，stdout（打印到shell）作为输出。这些输入输出通常为文本流（即字节序列），而我们可以改变这些流的方向（即重定向），例如将某个程序的输出保存到文件：echo hello > hello.txt ，或者从某个文件获得输入: ./a.out < test.txt。另外我们可以使用 2> 将标准错误（stderr）重定向到文件。

管道是将一个程序的输出衔接到另一个程序的输入（形成一个buffer，一个程序往里面写值，另一个程序从里面读值）

1.2 文件与伪文件系统

ls -al 显示所有文件及其权限（通常的 ls 命令会隐藏以 . 开头的文件）

在linux系统中，everything is file， 我们使用ls看到的目录、链接也是file的一种。如果没用目录文件的执行（x）权限，就无法对目录中的内容进行搜索（即无法索引），那么就找不到目录下的文件了，在这种情况下，即使该目录下的文件有读写或执行权限，也无法进行对应操作。

在根目录下，有部分文件不是与硬盘中的文件系统一一对应的，而是linux为了方便程序与系统的交互而设计的API，如获取并更改系统设置、查看进程、查看CPU信息等。例如 /proc/ 目录下存放当前所有进程的信息，这样我们就可以方便地与系统完成更好的交互。例如我们可以编写一个程序，通过读伪文件中的信息来打印当前的进程树、CPU状态等。

1.3 进程

ps, pstree, pidof, kill, jobs, &, bg, kill, nohup

命令行支持后台运行程序。我们先用python写一些长时间sleep的程序，并在命令结尾处加入 & 符号：python3 sleep.py & 。这样，sleep程序就在后台运行了，我们还可以通过Ctrl-z来切换程序到后台并暂停执行，而nohup命令可以在退出shell后让命令保持执行。

如果需要知道当前的shell下有哪些后台进程，可以使用 jobs 命令，具体进程操作请查看相应手册。

除shell之外，可以通过 ps 打印当前系统运行的多数进程信息， pstree 可以将这些进程以树的形式排列（被一个进程启动的称为这个进程的子进程）。kill命令可以给进程发送中止信息，或设置相应参数直接由操作系统中止进程（该操作为危险操作）。

1.4 信号机制

signal(7) - Linux manual page (man7.org)

我们可以写一个sigint.py来检测Ctrl+C对程序的中断机制（例如为什么退出不了vim）

  • print("\r{}".format(i), end=""): 这行代码打印计数器的值，但不换行（\r将光标移回行首）。end=""确保print函数在打印后不会添加默认的换行符。这样，每次循环时，计数器都会在同一行上更新。

我们可以看到，中断信号的处理并没有被写在循环内部，但在程序执行过程中，我们按下Ctrl+C，却得到了输出，并没有退出程序。

因此可以断定，中断信号的发出与处理是解释器的行为而非程序代码的行为，这有点类似与同步电路中的异步信号。（我们要特别考虑这些异步信号，因为它可能使程序进入非法状态）

再试着按下Ctrl+ \，观察到程序退出了，这是因为发送了SIGQUIT信号，而我们的程序中并没有对这个信号进行特殊处理，因此按默认功能退出。同理，kill 指令会生成 SIGTERM 信号，而这个信号同样可以被程序捕获，而做出相应的行为。

使用SIGKILL信号（由 kill -9 生成）可以强制退出程序，因为这个信号并不能被程序捕获，它直接由操作系统中止程序并不给任何保存文件的机会。注意这会留下孤儿进程。

2.2 程序的命令行参数

程序是可以接受命令行参数的，就像我们使用的命令那样。我们需要在C语言中做如下处理：

int main(int argc, char *argv[]) {...

  1. argc（Argument Count）
    • 这是一个整数，表示传递给 main 函数的命令行参数的数量（包括程序名称本身）。
    • 通常，argc 的值至少为1，因为第一个参数（argv[0]）是程序的名称。
  2. argv（Argument Vector）
    • 这是一个指向字符指针数组的指针，每个字符指针指向一个以null结尾的字符串（即C风格的字符串）。
    • argv[0] 通常是程序的名称。
    • argv[1] 到 argv[argc-1] 是传递给程序的命令行参数。

2.4 观察程序的系统调用

我们知道程序只参与计算（即与自己的内存、寄存器互动），是个无情的状态机，那么程序是如何停止的？

答案是，操作系统提供了一个接口：系统调用。系统调用是指令集中的一条指令。在执行这条指令时，系统的状态机将自身送给操作系统监管，操作系统将可能执行唤醒、结束、暂停进程等等一系列操作。

通过 strace 指令可以观察程序的系统调用。例如 strace ls > /dev/null 2> log ，strace的输出信息是默认发送到stderr的，因此我们使用 2> 将其重定向到 log 文件查看，并将ls的标准输出丢弃。我们从log文件中可以看到execve, mmap, access, pread64, write等等linux的系统调用函数。通过追踪程序的系统调用，我们可以更加清楚地了解到程序的行为过程。编译hello.c并查看系统调用可以发现，即使是最简单的Hello world程序，都要经过与操作系统的多次交互才能完成执行。

2.5 PATH

所有的命令都是程序的运行。当我们在shell中敲下 echo hello 时，shell是如何启动 echo 并传入参数的？

键入 echo $PATH 可以看到被 : 分隔的各个路径（fish中是用空格分隔）。这些路径表示了shell在运行命令时的搜索顺序。用 mv 指令将之前实现的fuck 的可执行文件移动到第一个路径下，这样我们可以直接将 fuck 当作命令使用。这样我们就知道了命令（程序）是如何被执行的。

2.6 使用gdb调试程序

在debug时，使用print大法是简便但是效率低下的，因此需要使用gdb完成程序的调试。调试前通常在Makefile文件中添加debug项并使用make启动以节省输入指令的时间。在使用gdb前，我们通常会使用 gcc -g 编译程序以留下源代码。用gdb运行程序，并layout src或其他信息以供参考。

  • list (l): 显示源代码，默认显示接下来的10行。你可以指定一个函数名或行号来查看特定的代码。
  • break (b): 设置断点。例如，b main会在main函数处设置一个断点。
  • run (r): 运行程序，直到遇到断点或程序结束。
  • next (n): 执行下一行代码，但不进入函数内部。
  • step (s): 执行下一行代码，如果下一行是函数调用，则进入该函数。
  • continue (c): 继续执行程序，直到遇到下一个断点或程序结束。
  • print (p): 打印变量的值。例如，p variable_name。
  • info breakpoints (info b): 列出所有的断点。
  • delete breakpoints (d): 删除断点。例如，d 1会删除编号为1的断点。
  • quit (q): 退出GDB。

除了这些简便的操作以外，我们还可以编写调试脚本以提高调试效率。调试脚本的编写方法将在以后的博客中发布。

4. 版本控制(git)

版本控制在项目管理和多人协作时非常有用

一个很好的git intro：Version Control (Git) · Missing Semester (mit.edu)

4.1 树形文件存储

采用链接机制，文件夹（tree）和文件（blob），tree可以包含tree和blob。这些关系信息被以哈希值表示。

  1. 文件内容（Blob）：当你将文件添加到 Git 仓库时，Git 会计算该文件的 SHA-1 哈希值，并将文件内容存储为一个 blob 对象。这个 blob 对象由其内容及其对应的 SHA-1 哈希值唯一标识。
  2. 目录结构（Tree）：Git 使用树对象来表示目录结构。一个树对象包含了对其他树对象或 blob 对象的引用，以及这些对象在目录结构中的路径。每个树对象也有一个唯一的 SHA-1 哈希值。
  3. 提交（Commit）：一个提交对象包含了关于仓库当前状态的信息，包括一个指向根树对象的指针、一个指向父提交的指针（或多个指针，对于合并提交）、作者和提交者信息、提交消息等。每个提交也有一个唯一的 SHA-1 哈希值。

4.2 快照机制

git使用快照（snapshot）来保存每个版本的提交，每个提交有一个或多个父提交（分支的合并）。而一个父提交也可能有多个子提交（分支）整个提交结构是一个有向无环图。

4.3 简单的git命令

  • 初始化仓库 git init
  • 添加文件到暂存区 git add
  • 查看修改内容 git diff
  • 查看暂存区 git status
  • 提交（暂存区）更改到HEAD指向的分支 git commit ，并写更改文档
  • 查看历史记录 git log
  • 回溯到某个快照 git checkout （这会导致未快照的本地文件被破坏）

4.4 分支与合并

  • 查看所有分支 git branch

  • 分支流程：

    • 在head处创建一个新的索引
    • 将head指向新的索引，这样提交时就默认更新head指向的这个索引

  • 分支合并：git merge, git mergetool ，具体使用方法参考文档

4.5 git远程仓库

  • 获取远程仓库的副本，并用其初始化本地仓库 git clone
  • 连接本地项目至远程仓库 git remote add
  • 推送本地更改到远程仓库 git push
  • 查看所有该项目的远程仓库 git remote
  • 将远程仓库的更新同步至本地 git fetch

4.6 一些惯例

  • master：代码中的主开发分支，通常意味着最新版本，可以视为指向最后一次提交的指针，有时候也为main
  • HEAD：特殊引用，指向正在查看的提交
  • origin：如果只有一个远程仓库，那么通常叫origin
